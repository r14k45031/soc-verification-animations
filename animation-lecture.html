Formal Modeling & Property Specification

本章核心問題：如何把硬體設計和它應該滿足的性質，都用數學精確描述，讓電腦自動驗證？

學習路徑：Kripke 建模 → 計算樹 → 時序邏輯 → Safety/Liveness → Assertion 驗證 → Explicit/Symbolic 技術

Formal 的優勢

  • 數學確定性（所有輸入 × 無限時間）
  • 產生最短 counterexample
  • 自動化，不需人工測資

Formal 的限制

  • State explosion 問題
  • Proof 可能因複雜度中斷
  • 性質需人工撰寫

② Model Checking 三步驟

Model Checking 是自動驗證「有限狀態並行系統」的技術，核心流程分三步：

正式定義：令 M 為並行系統的狀態轉移圖，f 為時序邏輯規格。Model Checking 找出 M 中所有滿足 f 的狀態 s。

為什麼不用 RTL 直接驗證？

RTL 有語言差異和語法細節，驗證工具難以統一處理。需要語言無關的數學抽象（Kripke Structure）作為統一基礎。

Formal vs Simulation

Simulation：跑什麼就驗什麼，無法保證完整性。Formal：窮舉所有輸入 × 無限時間，具數學確定性。

③ Kripke Structure — 形式化建模

M = (S, S₀, R, L)　　AP = {a, b, c}
符號名稱說明為什麼需要？
S有限狀態集電路 n 個暫存器 → 2ⁿ 個狀態系統所有可能的「時刻快照」
S₀ ⊆ S初始狀態集系統啟動時的狀態定義驗證起始點
R ⊆ S×S全射轉移關係每個狀態一定有後繼讓「無限執行」有意義
L: S→2^AP標記函數每狀態標記成立的命題連接狀態與性質的橋樑

④ 計算樹（Computation Tree）

把 Kripke Structure 從初始狀態 s₀ 沿所有可能轉移路徑無限展開，得到「無限計算樹」。每條路徑代表一個可能的執行序列。

為什麼需要計算樹？

單純的 Kripke STG 只能看「此刻的狀態關係」。計算樹把時間維度展開，讓我們能表達「沿執行路徑的時序性質」。

需要兩層量詞

Path quantifier（A/E）：選哪些路徑
State quantifier（G/F/X/U）：在路徑上描述時間行為

⑤ CTL* / CTL / LTL 時序邏輯

切換算子：
公式合法？說明
AG(p ⇒ EF q)CTL ✓每個 state quantifier 前有 A 或 E
AGF pCTL ✗G 和 F 之間沒有 A/E
AGF(p)LTL ✓= A(GF(p))，只有一層 A 前綴
AGAF(p)LTL ✗兩層 A，是 CTL 才能表達

⑥ Safety vs Liveness

Safety（不變性）

「好事永遠成立 / 壞事永不發生」

AG p ≣ ¬EF ¬p

反例：有限長路徑（某時間點 p 為 false）

Liveness（存活性）

「好事最終會發生 / 壞事不會永遠持續」

AF p ≣ ¬EG ¬p

反例：帶環 Lasso 路徑（環上全是 ¬p）

Liveness 無法用模擬驗證：Safety 反例有限長，模擬可抓到；Liveness 反例是「無限執行路徑中 p 從不出現」，模擬再長也無法確認。只有形式化方法才能找到 Lasso，證明違反。

⑦ Assertion-Based Verification & PSL/SVA

因為 CTL/LTL 對工程師不直觀，1990s 末開發出更易學的性質規格語言：PSL（IEEE 1850）和 SVA（SystemVerilog Assertions）。

assert_always(p) ≣ AG(p) ≣ ¬EF(¬p)
PSL / SVA含義等價 CTL
assert_never(p)p 永遠不成立AG(¬p)
assert_always(p)p 永遠成立AG(p)
assert_next(req, ack)req 後下一拍 ack 成立AG(req⇒AX ack)
req |-> ##[1:2] ackreq 後 1-2 拍內 ackSafety property

超過 90% 的工業界驗證性質都是 Safety（invariance）性質，可統一寫成 assert_always(p) 的形式。Safety 反例有限長、工具驗證快、模擬也能幫助驗證。

⑧ Explicit vs Symbolic Model Checking

Explicit Model Checking

  • 逐一枚舉每個狀態，存入記憶體
  • DFS / BFS 遍歷狀態空間
  • 直觀易實作
  • 記憶體需求 O(2ⁿ) → 指數爆炸

Symbolic (Implicit) Model Checking

  • 用 Boolean function 符號化表示狀態集合
  • BDD / SAT 等資料結構
  • 一次操作整批狀態
  • 大小通常 O(n) 到 O(n²)
暫存器位元數： 10 bits → 1,024 狀態

Symbolic 精髓：把「一批狀態」壓縮成「一個 Boolean function」。BDD 讓這個函數的 AND/OR/∃x 消去操作都能高效完成，一次計算整批狀態的 image，突破 State Explosion。
