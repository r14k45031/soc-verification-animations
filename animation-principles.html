🔬 Formal Modeling — 原理直覺講解（Why it works）

SoC Verification｜從「為什麼」的角度深入理解每個設計決策

① Kripke Structure — 為什麼要這樣建模？

核心問題：一個硬體電路有那麼多行為，要怎麼讓電腦「看得懂」並加以驗證？

驗證的本質是：對系統所有可能的執行，確認某個性質永遠成立。但「所有可能的執行」要怎麼表達？答案就是 Kripke Structure。

直覺：把電路想像成一張地圖

電路在任何時刻都有一個「當前狀態」（所有暫存器的值）。時脈到來時，電路根據當前狀態與輸入，跳到下一個狀態。這整個過程就是一張有方向的地圖——Kripke Structure。

為什麼不直接用 RTL code？

RTL 有語法、時序細節、不同語言寫法。Kripke Structure 是語言無關的數學抽象，讓所有驗證算法都建立在同一基礎上。

為什麼 R 必須是 Total（全射）？

若某狀態沒有後繼，系統就「卡死」了（deadlock）。要讓「無限執行」有意義，每個狀態都必須能繼續運行，即使是自我回路也算。

一句話記住：Kripke Structure 就是「有標記的無限狀態轉移圖」。標記告訴驗證器每個狀態有什麼性質，讓它能問「在所有可達狀態中，p 永遠成立嗎？」

② 時序邏輯 — 為什麼需要「兩層」量詞？

核心問題：一般邏輯（AND/OR/NOT）只能描述「此刻」的性質，但硬體需要描述「跨時間」的行為，怎麼辦？

考慮 arbiter 電路：「req 進來之後，ack 最終一定會出現」。普通布林邏輯完全無法表達——它只能描述單一時間點的組合邏輯關係。我們需要能描述「隨時間演進的行為」的語言。

為什麼需要「兩層」量詞（Path + State）？

選一個範例：

CTL* → CTL：為什麼要限制？

CTL* 表達力最強但工具開發困難。CTL 要求每個 state quantifier 前必須有 A/E，犧牲少量表達力，換來多項式時間的驗證算法。

CTL vs LTL 本質差異

CTL 是 branching-time（計算樹），可混合 A/E。LTL 是 linear-time（單一路徑），只有一層路徑前綴。兩者不互相包含。

兩層量詞的直覺：「Path quantifier（A/E）」選擇看哪些路徑，「State quantifier（G/F/X/U）」在選定的路徑上描述時間行為。少了任何一層，都無法完整表達並行系統的時序性質。

③ Safety vs Liveness — 本質差異在哪裡？

核心問題：都是「性質」，Safety 和 Liveness 為什麼驗證方式完全不同？

從「如何被違反」來理解本質差異

Safety 和 Liveness 的根本區別不在於「說了什麼」，而在於「反例（counterexample）長什麼樣子」。

為什麼 Liveness 不能用模擬來驗證？

Safety 被違反 → 在某個有限時間點 p 變成 false → 模擬跑到那個時間點就能抓到。

Liveness 被違反 → p 永遠不成立，這需要模擬無限長的時間才能確認！模擬再長也無法保證「未來某天 p 一定會出現」。

Liveness 違反 = 找到一個 Lasso（有環的執行路徑），且環上所有狀態都滿足 ¬p。因為有環就代表系統可以永遠在那個 loop 裡轉，p 永遠不成立。有限狀態系統中，「無限執行」一定包含一個環。

90% 硬體性質都是 Safety 的原因：Safety 反例有限長、能被模擬抓到、工具驗證更快。Liveness 性質更難，通常在設計高層確認（如協議本身保證不死鎖）。

④ Explicit vs Symbolic — 為什麼 Symbolic 能贏？

核心問題：狀態太多，明明記憶體放不下，Symbolic 為什麼能繞過這個問題？

先感受 State Explosion 有多嚴重

暫存器位元數： 10 bits → 1,024 狀態

Symbolic 的根本原理：把「一批狀態」包成一個數學物件

Explicit 方法把每個狀態都存成一筆記錄，100 萬個狀態就要 100 萬筆。Symbolic 方法則用一個 Boolean function 表示整批狀態：

Reached = {s₀, s₁, s₃, s₇, ...} → 用一個 BDD（Boolean function）表示整個集合

Symbolic 的核心操作：Image computation

從當前可達狀態集合，一步計算所有下一步可達的狀態集合——用 BDD 的 AND/OR/量詞消去完成，一次操作整批狀態。

Symbolic 的限制

BDD 大小仍取決於 variable ordering 與電路結構。乘法器等算術電路的 BDD 仍是指數大，這時需要 SAT-based 方法。

一句話記住 Symbolic 精髓：Explicit 是「一個一個數格子」，Symbolic 是「用一條數學式描述所有格子」。BDD 讓這條數學式的操作（AND、OR、∃x 消去）都能高效完成。
